# Feature Specification: キーワード抽出とカバー率検証

**Feature Branch**: `071-keyword-coverage-validation`
**Created**: 2026-03-28
**Status**: Draft
**Input**: User description: "対話生成の品質向上のため、原文から重要キーワードを抽出し、生成された対話がキーワードをカバーしているか検証する機能を実装する"
**Parent Issue**: #64
**Depends on**: #70

## 概要

対話生成の品質向上のため、以下の2つの機能を実装する:

1. **キーワード抽出**: 原文セクションから重要キーワードを抽出
2. **カバー率検証**: 生成された対話XMLがキーワードをカバーしているか検証

## User Scenarios & Testing

### User Story 1 - キーワード抽出 (Priority: P1)

ユーザーとして、原文セクションから重要なキーワードを自動抽出したい。これにより、対話生成後に内容の網羅性を検証できる。

**Why this priority**: キーワード抽出はカバー率検証の前提条件であり、この機能なしには検証が成り立たない。

**Independent Test**: 原文テキストを入力として与え、キーワードリストが出力されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 原文セクションのテキストがある, **When** キーワード抽出を実行する, **Then** 重要キーワードがカンマ区切りリストとして出力される
2. **Given** 固有名詞、専門用語、数値を含むテキスト, **When** キーワード抽出を実行する, **Then** これらの要素がキーワードとして抽出される
3. **Given** 空のテキスト, **When** キーワード抽出を実行する, **Then** 空のキーワードリストが返される

---

### User Story 2 - カバー率検証 (Priority: P1)

ユーザーとして、生成された対話XMLが抽出したキーワードをどの程度カバーしているか確認したい。これにより、対話の品質を定量的に評価できる。

**Why this priority**: カバー率検証は品質保証の核心機能であり、キーワード抽出と同等に重要。

**Independent Test**: キーワードリストと対話XMLを入力として与え、カバー率と未カバーキーワードが出力されることを確認できる。

**Acceptance Scenarios**:

1. **Given** キーワードリストと対話XMLがある, **When** カバー率検証を実行する, **Then** カバー率（0.0〜1.0）と未カバーキーワードリストが出力される
2. **Given** 全キーワードが対話に含まれる, **When** カバー率検証を実行する, **Then** カバー率は1.0で未カバーリストは空になる
3. **Given** キーワードが一つも対話に含まれない, **When** カバー率検証を実行する, **Then** カバー率は0.0で全キーワードが未カバーリストに含まれる

---

### User Story 3 - 検証結果の出力 (Priority: P2)

ユーザーとして、カバー率検証の結果をJSON形式で取得したい。これにより、他のツールやログ分析で活用できる。

**Why this priority**: 結果の永続化は品質改善のPDCAサイクルに必要だが、コア機能の後に実装可能。

**Independent Test**: 検証実行後にJSON形式の出力が得られることを確認できる。

**Acceptance Scenarios**:

1. **Given** カバー率検証が完了した, **When** 結果を取得する, **Then** total_keywords, covered_keywords, coverage_rate, missing_keywordsを含むJSONが出力される

---

### Edge Cases

- 空のテキストからキーワード抽出を試みた場合 → 空リストを返す
- 空のキーワードリストでカバー率検証を実行した場合 → カバー率1.0（全てカバー済みと見なす）
- 対話XMLが空または不正な形式の場合 → カバー率0.0、全キーワードが未カバー
- キーワードに部分一致のみがある場合 → 完全一致のみをカバーと判定

## Requirements

### Functional Requirements

- **FR-001**: システムは原文テキストからキーワードを抽出できなければならない
- **FR-002**: キーワード抽出では固有名詞（会社名、人名、製品名）を識別できなければならない
- **FR-003**: キーワード抽出では専門用語・概念を識別できなければならない
- **FR-004**: キーワード抽出では具体的な数値・日付を識別できなければならない
- **FR-005**: キーワード抽出では重要なアクション・イベントを識別できなければならない
- **FR-006**: システムは対話XMLに対してキーワードのカバー率を計算できなければならない
- **FR-007**: カバー率検証は文字列マッチングで実装し、LLMを使用しない
- **FR-008**: 検証結果はJSON形式で出力できなければならない
- **FR-009**: 検証結果には総キーワード数、カバー数、カバー率、未カバーキーワードリストを含めなければならない

### Key Entities

- **キーワードリスト**: 抽出されたキーワードの集合。各キーワードは文字列。
- **検証結果**: カバー率検証の出力。総数、カバー数、カバー率、未カバーリストを含む。
- **原文セクション**: キーワード抽出の入力となるテキスト。
- **対話XML**: カバー率検証の対象となる生成済み対話データ。

## Success Criteria

### Measurable Outcomes

- **SC-001**: キーワード抽出は1セクションあたり5秒以内に完了する
- **SC-002**: 抽出されるキーワードは原文の重要概念の80%以上をカバーする
- **SC-003**: カバー率検証は1対話あたり1秒以内に完了する（LLM不使用のため）
- **SC-004**: カバー率の計算精度は100%である（文字列マッチングによる決定論的処理）
- **SC-005**: 検証結果のJSON出力は定義されたスキーマに100%準拠する

## Assumptions

- 既存の対話生成パイプライン（#70で外部化されたプロンプト）が利用可能
- LLMはキーワード抽出にのみ使用し、カバー率検証には使用しない
- キーワードの一致判定は完全一致（大文字小文字は区別しない）で行う
- 対話XMLは既存の形式（`<dialogue>`タグ構造）に従う
